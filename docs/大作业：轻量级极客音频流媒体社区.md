# 软件开发系统综合能力实践 大作业

# echo：轻量级极客音频流媒体社区

## 一、 核心选题定位

直白地说：“在线版且轻量版网易云音乐”

本项目旨在打破传统“增删改查”管理后台的刻板套路，构建一个以**多媒体流式传输**与**单曲垂直互动**为核心的轻量级极客音频分享社区。项目依托 Python (Django) 和 SQLite 构建稳健的底层数据与文件安全防线，前端采用 Tailwind CSS 打造现代化的视觉 UI，并深度整合 HTMX 与原生 JavaScript 技术，实现了“**页面无缝局部切换、底层音乐平滑不中断**”的类 SPA（单页应用）沉浸式体验，在确保敏捷开发按期交付的同时，极致拉满了项目的工程含金量与视觉张力。

---

## 二、 技术栈选型与依据

这是一套“底层稳健 + 前端前卫”的差异化技术组合：

- **后端引擎：Python + Django**
  - **职责：** 核心业务路由、ORM 映射、底层文件流处理与安全校验。
  - **依据：** 完美契合课程核心考核要求，MVC/MVT 架构清晰。
- **数据底座：SQLite3**
  - **职责：** 关系型数据持久化存储。
  - **依据：** 零环境依赖，确保项目提交后在任何机器上均可“一键无错运行”，稳拿环境部署基础分。
- **UI 视觉架构：Tailwind CSS (`django-tailwind`)**
  - **职责：** 响应式布局与极客风 UI 绘制。
  - **依据：** 替代老旧的 Bootstrap，通过原子化 CSS 实现高度定制的现代界面，拉开与同班同学的视觉差距。
- **前端动态交互：原生 JavaScript + HTMX**
  - **职责：** 操控音频播放器状态、实现无刷新局部页面更新。
  - **依据：** 摒弃臃肿的 Vue/React 部署深坑，以最低的成本实现接近 SPA（单页应用）的高级用户体验。

## 三、 需求规格说明书 (PRD)

根据敏捷开发理念，系统需求分为核心功能与非核心优化项。

### 1. 核心功能模块 (FR)

- **安全媒体库 (Audio Hub)：**
  - 支持用户（Uploader）上传 `.mp3` 与 `.wav` 音频文件及可选封面图。
  - 系统需在底层完成文件大小拦截（<10MB）与 UUID 安全重命名，防止覆盖与注入。
  - 支持真实播放量（Play Count）的动态统计与按时间倒序的列表展示。
- **现代播放器 (Modern Player)：**
  - 页面底部提供常驻的自定义音频控制条。
  - 支持基础的播放、暂停、进度条拖拽控制。
- **单曲评论区 (Track Comments)：**
  - 为每首独立音频提供专属的下沉式评论流。
  - 支持用户发布评论，评论按时间倒序（最新置顶）自动刷新显示。

### 2. 用户体验与安全需求 (NFR)

- **连续性体验：** 用户在进行浏览评论、切换列表等操作时，底部的音频播放器尽量保持状态不中断。
- **数据级联保护：** 若音频或用户被删除，相关联的底层文件与评论需实现防雪崩式的自动级联清理 (`CASCADE`)。

## 四、 敏捷开发与编程规划 (Agile Roadmap)

为确保项目按时交付并规避“烂尾”风险，开发排期分为四个递进阶段（Phase）。**强烈建议严格按顺序执行，切勿跨阶段开发。**

### 🚩 Phase 1：MVP 核心底座（保底及格线：优先完成）

- **目标：** 打通“上传 -> 数据库 -> 播放”的主干道。
- **开发任务：**
  1. 初始化 Django 项目，配置 SQLite 与 `MEDIA_ROOT` 路径。
  2. 集成 `django-tailwind` 构建基础 HTML 骨架。
  3. 编写 `AudioTrack` 核心模型，执行 `makemigrations` 与 `migrate`。
  4. 开发音频上传视图（View）与表单（Form），并完成本地文件保存测试。
  5. 开发主页列表视图，使用原生 `<audio>` 标签实现最基础的播放功能。

### 🚩 Phase 2：单曲互动社区（良好线：体现完整 CRUD）

- **目标：** 实现数据关联，完成完整的业务闭环。
- **开发任务：**
  1. 编写 `Comment` 模型，建立与 `AudioTrack` 和 `User` 的外键关联。
  2. 开发音频详情页视图（DetailView），渲染单曲信息与下方评论列表。
  3. 完成“提交评论”的表单处理逻辑，并测试数据回显。
  4. 使用 Tailwind 深度美化“音频卡片”与“评论流” UI。

### 🚩 Phase 3：局部刷新增强（高分线：技术炫技）

- **目标：** 攻克“页面跳转导致音乐中断”的痛点。
- **开发任务：**
  1. 引入 HTMX 脚本或编写原生 JS `fetch` 请求。
  2. 将“评论提交”改造为局部刷新：提交后不刷新整页，仅向评论列表中动态追加新数据。
  3. （进阶挑战）锁定底部播放器 DOM，尝试将页面主内容区的跳转全部改为局部 HTML 片段替换。

### 🚩 Phase 4：数据驱动与细节打磨（满分线：工程级验收）

- **目标：** 修复边界漏洞，完善产品细节。
- **开发任务：**
  1. 激活 `play_count` 字段，绑定前端播放按钮的点击事件，实现播放量累加。
  2. 处理上传报错反馈（如文件超过 10MB 时在前端显示优雅的提示）。
  3. 配置 404 错误页，梳理项目演示文档与源码打包结构。
